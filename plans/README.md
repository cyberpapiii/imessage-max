## Findings considered and rejected

Recorded so future audits don't re-litigate:

- **Force-unwrap pattern in Send.swift:318 / SendResolution.swift:83,299** (`text?.isEmpty == false ? text! : ...`): technically safe under the guarding condition; style-only, not plan-worthy.
- **Session ID transmitted over plaintext HTTP**: by-design for a loopback-only transport; revisit only if network exposure ever becomes supported.
- **Missing HTTP authentication layer**: real, but an architectural decision belonging to the Trustworthy Core track, not a bolt-on fix. Plan 005 closes the accidental-exposure path; deliberate network exposure should arrive together with an auth design.
- **`LIKE '%term%'` full-table scans in search/find_chat** (`SearchInternals.swift:106-137`, `FindChat.swift:407,432`): real cost on large chat.db files, but the obvious fix (FTS5) is infeasible in place — chat.db is Apple's database, opened read-only by design. A sidecar FTS index is a direction-level feature (sync/staleness tradeoffs); raise it when search latency is actually hurting.
- **Unstructured `Task` around the routing continuation** (`HTTPTransport.swift:271-287`): the pending-request timeout (~line 514) already bounds the leak; hardening is possible but low-value relative to risk of touching transport code.
- **SwiftLint/swift-format setup**: fine idea, low value for a single-maintainer repo right now; revisit if contributors arrive.
- **Inconsistent tool error patterns** (`Result` returns vs `throws` across tools): real debt, but mechanical and low-impact; fold into a future tool-layer consolidation rather than a standalone pass.
- **FindChat migration onto ChatSummaryQueries** (2026-06-11): rejected — `find_chat`'s per-candidate enrichment is bounded by small result counts (`targetCount ≤ ~5`), so the batching win is negligible against the refactor risk. Revisit only with reported find_chat latency.
